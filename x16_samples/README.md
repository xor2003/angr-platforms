# x16_samples

Small real-mode sample corpus for decompiler and loader testing.

Quick start:
- From the repo root, run `./scripts/build_x16_samples.sh`
- Then run `../venv/bin/python -m pytest -q tests/test_x86_16_sample_matrix.py`

Artifacts generated by `build_matrix.sh`:
- MSC 5.1 `.EXE` variants built from `IDEMO.C`
- matching `.COD` listings copied per variant
- UASM `.COM` variants built from tiny-model assembly
- `matrix_manifest.json` describing the generated outputs

Notes:
- The packaged `LINK.EXE` in this toolchain does not accept the documented tiny-model `/T` or `/TINY` options, so compiler-generated `.COM` binaries are not currently emitted through MSC in this environment.
- To keep `.COM` coverage in the corpus, `ICOMDO.ASM` and `ICOMBI.ASM` are assembled directly with UASM to flat binaries.
- `build_matrix.sh` expects the DOS toolchain checkout at `/home/xor/games/f15se2-re` by default. Override it with `X16_TOOLCHAIN_ROOT=/path/to/f15se2-re ./scripts/build_x16_samples.sh`.
- `build_matrix.sh` creates compatibility copies for `MLIBCE.LIB` and `LLIBCE.LIB` from `MLIBCR.LIB` and `LLIBCR.LIB` if they are missing. Medium, large, and huge MSC builds depend on those names in this environment.
- The script also removes DOSBox byproducts such as `*.dos.log`, `*.emu.log`, `*.dosbuild.*`, `*.rsp`, `IDEMO.COD`, and `LOG.TXT` after each run so the directory returns to the committed corpus layout.
- `0x417` in the recovered BIOS sample corresponds to `0x40:0x17`, the BIOS Data Area keyboard flag byte 0.
